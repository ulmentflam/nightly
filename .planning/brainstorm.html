Nightly.

A continuously-running, harness-agnostic coding agent that picks up where humans left off, ranks its own work, and lands clean, review-shaped changes by morning — pull requests when there's a remote, local proposals when there isn't — then briefs you in HTML when you tell it to wrap up.

01 · Thesis

An agent shaped around overnight, not turn-by-turn.

Every shipping autonomous coding agent today is either tethered to a human reviewer turn-by-turn (Cursor, Claude Code interactive) or single-shot on an issue handed to it by name (Sweep, Copilot coding agent, Devin). Nightly is neither. It is a session-shaped agent: you start it, it works for hours or days, it self-directs across many issues and PRs, and it lands cleanly when you ask it to conclude.

The product we are designing is not "a smarter LLM." It is a thin, opinionated harness that wraps existing CLI agents (Claude Code, Cursor, Codex, Antigravity, opencode) and gives them three things they don't already have together: a persistent overnight plan, autonomous task selection across an open backlog, and a graceful drain protocol.

North star

You hand Nightly a repo and a tea. It hands you back, hours later, a stack of branch-isolated PRs and an HTML briefing with everything it touched, what it skipped and why, and what it would do next.

First principles

  1. Branch-isolated, PR-shaped deliverables. Not a dashboard, not a chat log. Every task ends as a reviewable artifact on an isolated branch. If the repo has a GitHub remote, that artifact is a real pull request. If it doesn't — local repos, GitLab, self-hosted, air-gapped — it's a local proposal of the same shape (title · summary · diff · test plan · uncertainty block) that surfaces in the morning briefing as if it were a PR.
  2. Surface uncertainty louder than progress. Devin's well-known failure was emitting a destructive migration without flagging risk. Nightly's PR (or proposal) body has a mandatory "Things I'm not sure about / Things that could break" block; absent or empty is a check failure.
  3. Always advance, never block. Approvals, missing credentials, and ambiguous requirements are recorded and routed around — never waited on. Shutdown is the same shape: a cooperative two-threshold drain (nudge → hard), never SIGKILL. Half-finished work parks on nightly/wip-<run>, never main. The run is monotonic — every iteration makes forward progress, even if "forward" means parking the current task and picking the next.
  4. Interactive-first; subscription-native. Nightly defaults to launching from inside an interactive session of a host you already use — Claude Code, Codex, Cursor, opencode, Antigravity — implemented natively as a Skill, slash command, or agent. The host session is the long-running container; Nightly's loop runs inside it and dispatches specialists via each host's native sub-agent mechanism (Task tool · MCP · background agents). Because the host is already authenticated, Nightly inherits its subscription (Claude Pro/Max, ChatGPT Plus/Pro, Cursor account, Google OAuth) and never asks for an API token. A headless mode for cron / CI exists but is opt-in.
  5. Memory is local; version control is yours. Two folders organize artifacts by audience: .nightly/ for agent runtime state (plans, run logs, the repo atlas) and .planning/ for human-curated planning (this document, RFCs, design notes). Nightly never edits your .gitignore — what to commit, what to ignore, even whether to use those exact paths is entirely the user's call. The folder names are defaults; they're configurable in config.yml.
Why "Nightly" Most code agents are designed for a turn — one ask, one reply. The few designed for a session are either tethered (Cursor) or fire-and-forget (Devin issue → PR). Nightly is the third shape: session-shaped, self-directed, drainable. It maps to a human cadence — kick it off before bed, review in the morning — that the existing tools don't fit.

02 · Prior art & what to steal

Six ideas worth lifting whole.

Deep-read of the autonomous SWE space (Devin, OpenHands, SWE-agent, Sweep, AutoCodeRover, Copilot coding agent, Factory Droids, Replit Agent, Amp) surfaces a small set of design ideas that recur in the systems that actually work — and a larger set of failure modes from systems that didn't.

Steal · 01 · Wiki-as-memory (Devin)

Devin pre-computes a per-repo wiki (DeepWiki) — call graphs, module purposes, conventions — that is loaded into every session so the agent doesn't rediscover the codebase. Nightly should generate and refresh .nightly/atlas/ on every cold start.

Steal · 02 · Typed EventLog (OpenHands)

OpenHands persists every action and observation as a typed event in an append-only log. This makes pause / resume / replay essentially free. Nightly's orchestrator should adopt the same shape — every harness adapter emits into one normalized NightlyEvent stream, persisted as JSONL.

Steal · 03 · Agent-Computer Interface (SWE-agent)

Princeton's insight: lint-on-edit + paged-file-viewer + terse search took the same model from 2% to 12.5% on SWE-bench. Nightly should never let any harness write syntactically broken code, and should expose retrieval as paged, AST-aware search rather than raw grep.

Steal · 04 · AST-aware retrieval (Sweep + AutoCodeRover)

Chunk by function / class / module, not by token window. Expose structured queries: search_method("foo"), callers_of("bar"), defs_in_file(...). When tests fail, prime retrieval with spectrum-based fault localization.

Steal · 05 · Two-threshold cooperative drain (LangGraph / killswitch.md)

A nudge phase asks the agent to finalize and write the report. A hard phase revokes the write toolset entirely and forces emit-only. Never SIGKILL. Stash WIP to nightly/wip-<run-id> so resuming next session is trivial.

Steal · 06 · Antigravity's on-disk artifact contract

Antigravity persists task.md, implementation_plan.md, walkthrough.md per task in ~/.gemini/antigravity/brain/<GUID>/. This is the cleanest "agent → orchestrator" contract in the field. Nightly's own per-run folder should follow the same shape.

The gap nobody is filling — and the one that defines Nightly — is autonomous task selection. Every leading system waits to be told which issue to work on. — synthesis of prior art, 2026

03 · The Loop

One state machine. Three phases. Always drains.

Nightly runs a single outer loop that cycles through Plan → Execute → Report. The loop's source of truth for "what to do next" is a strict priority cascade that mirrors how a competent engineer would behave waking up to an unfamiliar repo.

The priority cascade for PICK

This is the heart of "autonomous." Every iteration, Nightly resolves the next unit of work by walking this cascade top-to-bottom and stopping at the first match:

  1. A previous-session plan exists and is unfinished. Resume it. Plans live at .nightly/plans/<date>-<slug>.md with explicit status. Honoring stale plans is the cheapest, highest-value behavior — the human already approved the scope.
  2. A task parked as blocked: approval has just been unblocked. Any of the three approval surfaces (per-op CLI, per-task label, repo policy) can flip a parked task to ready: retry. The cascade picks the unblocked task before reaching for fresh work — the human already started it.
  3. A previous-session summary names follow-up work. The briefing.html from the prior run includes a "next session would do" block. Treat each item like a queued plan.
  4. An accepted RFC in .planning/rfcs/ has unstarted work items. RFCs with front-matter status: accepted are explicitly-blessed scopes — higher signal than a random GitHub issue. Nightly parses the RFC's task list, picks the next unchecked item, and writes a scoped plan that cites the RFC verbatim.
  5. Open issues exist on the remote. Triage the backlog (see §06 Task Selection), pick the highest-ranked nightly-eligible issue, write a scoped plan, execute.
  6. Open PRs need CI rescue or review-comment cleanup. Nightly-authored PRs with red CI or a maintainer's review comment are higher priority than picking a fresh issue — finishing beats starting. (In non-remote mode this becomes: open nightly/* branches with failing local checks.)
  7. No backlog. Ideate. Run the proposer suite (dep drift, coverage gaps, doc-vs-code drift, TODO/FIXME density, flaky tests, lint debt, a11y, perf hotspots). Emit ranked draft issues, not PRs. Land them in .nightly/runs/<ts>/proposed/ (and on the remote if one exists), then loop to pick one if any meet the autonomy bar.

The cascade is intentionally written so the riskiest behavior — pure ideation — is the lowest priority. If there is any honest work to do, Nightly does that first.

04 · Repo contract

Two folders. Split by audience, not by commit policy.

Nightly uses two top-level folders in the working tree. They're separated by audience and lifecycle, not by version-control treatment. Nightly never edits your .gitignore and never assumes either folder is tracked. What to commit is your call — different users land in different places, and Nightly works either way.

.planning/ is for humans: brainstorms (like this document), RFCs, ADRs, design notes, repo conventions worth writing down. Long-lived, low-volume, worth reading in a year. .nightly/ is for the agent: plans, run logs, the repo atlas, summaries — the transient state of an autonomous session. High-volume, short-lived, worth reading for a day. Both are sensible defaults; the paths are configurable in config.yml (rename, move, point at docs/agent/ or .agent/ or whatever fits your repo).

Common configurations
  • Both ignored. Treat both folders as scratch — useful for solo experimentation or air-gapped runs.
  • Both committed. Full audit trail; the repo is the database. Useful for shared-team Nightly habits where reproducibility matters.
  • .planning committed, .nightly ignored (this brainstorm's author's preference). Share design intent; keep agent log piles out of code review.
  • .nightly committed, .planning ignored. Unusual but valid — useful when planning is done elsewhere (Linear, Notion) but a shared replay trail is desired.

.planning/ — human-curated

.planning/ # human-authored; design intent that outlives any session ├── brainstorm.html # this document ├── rfcs/ # proposals before they become code │ └── 001-multi-harness-adapter.md ├── decisions/ # short ADRs — what we chose, why ├── conventions.md # house style, branching, review norms └── nightly.scope.md # what Nightly is and isn't allowed to do

.nightly/ — agent runtime

.nightly/ # agent-emitted; renewable, high-volume, easy to regenerate ├── config.yml # harnesses, models, guardrails, folder paths ├── atlas/ # Devin-style "wiki" — refreshed on cold start │ ├── overview.md # architecture, conventions, build commands │ ├── modules.json # module graph, public APIs │ └── conventions.md # inferred conventions (curated lives in .planning/) ├── plans/ # per-task scoped plans, append-only │ ├── 2026-05-19-fix-flaky-auth.md │ └── 2026-05-20-add-retry-budget.md ├── runs/ # one folder per session │ └── 2026-05-20T22-14Z/ │ ├── session.jsonl # normalized event log (OpenHands-style) │ ├── summary.md # human-readable narrative │ ├── briefing.html # the morning report (rendered on drain) │ ├── tasks/ # one folder per task this session │ │ └── 0001-add-retry-budget/ │ │ ├── plan.md │ │ ├── walkthrough.md # Antigravity-style │ │ ├── diff.patch │ │ ├── proposal.md # PR-shaped: title · summary · risks · test plan │ │ └── uncertainty.md # required, validated before "ready" │ └── proposed/ # ranked draft issues, not yet executed ├── memory/ # long-lived agent memory across sessions │ ├── lessons.md # "things I got wrong" — write rarely, read always │ └── conventions.local.md # learned-not-told repo conventions └── prompts/ # reusable system prompts per harness

How the two folders interact

The boundary is real but porous in one direction: Nightly reads from .planning/ and selectively pulls content into .nightly/ as runtime context. The reverse never happens automatically — Nightly never writes to .planning/. Human-authored artifacts stay human-owned; promotion of agent-emitted drafts into .planning/ requires a deliberate human action.

Three ingestion points:

  • At cold start. Nightly scans .planning/ and folds relevant content into .nightly/:
    • .planning/conventions.md → seeds .nightly/atlas/conventions.md (human-curated conventions are loaded first; inferred ones layer on top and never overwrite).
    • .planning/nightly.scope.md → folded into the system prompt as hard scope limits (what Nightly may and may not touch).
    • .planning/rfcs/*.md with status accepted → indexed for retrieval so harnesses can cite them.
    • .planning/decisions/*.md (ADRs) → indexed; surfaced when a task touches files the ADR scoped.
    • .planning/brainstorm.html and untagged notes → indexed for retrieval but flagged as aspirational, not authoritative.
  • At task pick. When the chosen task references an RFC, ADR, or convention by name (or when the atlas links the touched code path to one), the referenced document is copied verbatim into that task's plan.md context section, so the harness sees the same words the human wrote — no summarization, no rephrasing.
  • At drain. If Nightly identifies work that belongs in .planning/ rather than as a PR — a proposed RFC for a larger refactor, a new ADR for a decision it had to make, an addition to conventions.md — it writes the draft to .nightly/runs/<ts>/proposed/planning/ and surfaces it in the briefing's "Things to approve" section. It does not move it into .planning/. The human accepts by moving the file themselves (or via a future nightly promote-rfc <id> helper).

Nightly maintains .nightly/atlas/planning-index.json — a mapping from .planning/ source files to the .nightly/ artifacts they seeded, with content hashes. On the next cold start, changed sources trigger a re-ingest of only the affected downstream artifacts. The human edits .planning/conventions.md, and the next Nightly run sees the change automatically.

External context Nightly auto-detects

Nightly doesn't only read its own .planning/. It also pulls from established cross-tool conventions so users adopting Nightly don't have to rewrite their existing agent rules. Same one-way flow: read-only, ingested into .nightly/, never modified.

AGENTS.md · the cross-tool standard

AGENTS.md is now stewarded by the Linux Foundation's Agentic AI Foundation and adopted across Codex, Cursor, Antigravity, opencode, Copilot, Jules, Factory, Aider, Junie, Windsurf, and Amazon Q (~60k repos as of mid-2026). Nightly reads it natively as a first-class context source, not as a Nightly-specific config.

CLAUDE.md is also read, both because Claude Code is one of Nightly's primary harness backends and because real-world repos accumulate both files during the migration period. If both exist, AGENTS.md wins on conflict; CLAUDE.md fills any sections AGENTS.md doesn't cover.

Precedence mirrors Codex's lookup, walking root → working directory with closest-wins semantics for monorepos:

# .nightly/config.yml — context discovery (defaults shown)
context:
  rules:
    - AGENTS.override.md   # local-only overrides; typically untracked
    - AGENTS.md            # cross-tool standard — preferred
    - CLAUDE.md            # Anthropic's holdout; read as companion / fallback
    - .cursor/rules/**/*.mdc# Cursor rules — indexed if present

  planning:
    - .planning/           # Nightly's default — human-curated
    - specs/               # GitHub spec-kit (~88k stars)
    - docs/adr/            # Nygard ADRs
    - docs/rfc/            # RFCs
    - docs/design/         # long-form design docs
    - PLAN.md              # OpenAI cookbook ExecPlans
    - PLANS.md             # same, plural

  closest_wins: true          # nested AGENTS.md / CLAUDE.md override root
  trust:
    rules:    authoritative   # cannot be contradicted by inference
    planning: aspirational    # indexed, cited, but not binding

The discovery is order-sensitive — earlier entries win on conflict — and every detected source is logged in session.jsonl with a content hash, so a reviewer can see exactly which files shaped any given task. None of this is mandatory: a repo with only AGENTS.md and nothing in .planning/ works fine; a repo with only docs/adr/ works fine; a Nightly-native repo with .planning/ and no AGENTS.md works fine.

What Nightly does not do
  • Does not invent NIGHTLY.md. That mistake — every harness shipping its own rules-file format — was the reason AGENTS.md exists in the first place.
  • Does not write to AGENTS.md, CLAUDE.md, specs/, docs/, or any of the discovered paths. Those are human-owned.
  • Does not require the .planning/ folder name. Configure planning: however you like; .planning/ is just the default.

Why split this way

  1. Signal vs noise. Human-authored design (worth reading in a year) and agent-emitted runtime state (worth reading for a day) shouldn't share a folder. Mixing them buries the signal regardless of whether either is committed.
  2. Regeneration boundary. Everything in .nightly/ is, in principle, regenerable from the repo + .planning/ + a cold Nightly start. Everything in .planning/ is not. That boundary is more important than commit policy and survives any version-control choice.
  3. One-way trust. Pulling human-authored constraints into agent runtime is safe and desirable. Letting the agent rewrite human-authored intent is not. The asymmetric flow encodes that.
  4. Local-first. Nightly works on a fresh clone with no remote at all. The agent's state never depends on a server it can reach.

05 · Hosts & how Nightly plugs into each

One Nightly. Implemented natively inside every host you already use.

Nightly's default mode is to run inside an interactive session of a host you've already opened — Claude Code, Codex, Cursor, opencode, Antigravity. It's distributed as a thin, native integration for each: a Skill, a slash command, an agent registration. They all load the same core library — loop, priority cascade, drain, briefing renderer — and read/write the same on-disk artifact contract (.nightly/). The host session is the long-running container; Nightly orchestrates within it, dispatching specialist sub-agents through each host's native mechanism.

This design answers four things at once. Subscription auth — the host is already logged in, so Nightly inherits Claude Pro/Max, ChatGPT Plus/Pro, Cursor account, or Google OAuth without ever asking for a token. UX continuity — no new tool to learn; Nightly lives in the editor you already use. Sub-agent dispatch — every modern host has Task-tool / MCP / background-agent primitives, so Nightly reuses them instead of inventing its own. Context-window economics — sub-agents get their own context windows, so the host session isn't forced to hold every task's working memory.

Host Interactive launcher Sub-agent dispatch primitives Subscription source v1 priority
Claude Code .claude/skills/nightly/SKILL.md + /nightly Task tool (sub-agents with own context) · MCP · hooks Claude Pro · Max · Team primary
Codex CLI ~/.codex/skills/nightly/SKILL.md + slash command MCP client & server · codex exec sub-spawn · Seatbelt/Landlock ChatGPT Plus · Pro · Business · Edu · Enterprise primary
opencode .opencode/agents/nightly/ + slash command via SSE server Sub-agents · plugins · MCP opencode auth (per-provider, in auth.json) primary

What every host integration must implement

Each per-host integration is small — typically < 500 lines — because the loop logic lives in the shared core. The integration only has to handle five things: registering the launcher with the host, dispatching specialist sub-agents through the host's mechanism, exposing the host's session id, surfacing approval prompts through the host's native UI, and reporting subscription health.

Stack · Python 3.12+ with uv

Nightly is implemented in Python 3.12+, managed with uv. The core library and each host integration ship as separate uv-managed packages — nightly-core, nightly-host-claude, nightly-host-codex, and so on — in a single workspace.

Inner stack: asyncio throughout · httpx (HTTP) + httpx-sse (opencode's SSE event stream) · pydantic v2 (data models, config, event envelopes) · typer (CLI) · jinja2 (briefing.html renderer) · tree-sitter (atlas / AST-aware retrieval) · markdown-it-py (RFC and AGENTS.md ingestion). Dev loop: ruff (lint + format) · Pyrefly (types — Meta's Rust-backed checker) · pytest + pytest-asyncio, all wired behind a single Makefile (make check, make test, make type).

Why Python over TS: every host boundary is either REST, MCP, markdown, or subprocess — all of which Python handles natively. The Claude Agent SDK has first-class Python support; MCP servers are predominantly Python; AST tooling (tree-sitter, ast) and AGENTS.md / RFC parsing are more mature here. The one TS-leaning piece is opencode's first-party SDK, but its REST/SSE surface is trivially callable from httpx.

# nightly/hosts/contract.py
from abc import ABC, abstractmethod
from datetime import datetime
from typing import Literal
from pydantic import BaseModel

HostId = Literal["claude", "codex", "cursor", "opencode", "antigravity"]
SpecialistRole = Literal["implementer", "tester", "reviewer", "researcher"]


class SubAgentResult(BaseModel):
    role: SpecialistRole
    output: str
    tool_calls: list[dict]
    elapsed_ms: int


class AuthStatus(BaseModel):
    ok: bool
    plan: str | None = None
    expires_at: datetime | None = None


class NightlyHostIntegration(ABC):
    host_id: HostId

    # idempotent: install the launcher (Skill / command / agent file) into the host
    @abstractmethod
    async def install(self, scope: Literal["project", "user"]) -> None: ...

    # dispatch a specialist sub-agent through the host's native primitive
    @abstractmethod
    async def dispatch_sub_agent(
        self,
        *,
        role: SpecialistRole,
        prompt: str,
        cwd: str,                         # per-task git worktree
        allowed_tools: list[str] | None = None,
        timeout_s: float | None = None,
    ) -> SubAgentResult: ...

    # host's session id — recorded in session.jsonl for audit / replay
    @abstractmethod
    def session_id(self) -> str: ...

    # surface an approval question through the host's native UI
    @abstractmethod
    async def request_approval(self, q: str, choices: list[str]) -> str: ...

    # verify subscription is alive and won't expire mid-session
    @abstractmethod
    async def auth_status(self) -> AuthStatus: ...

The shared core calls dispatch_sub_agent whenever the loop delegates work to a specialist. Inside Claude Code that becomes a Task-tool invocation via the Claude Agent SDK; inside Codex it's an MCP call or a spawned codex exec; inside Cursor it can spawn a Background Agent. The core never knows which host it's in — only the integration does. That's the seam.

Per-host mapping notes

  • Claude Code — the natural primary. dispatchSubAgent maps to the Task tool; specialists run with their own sub-agent contexts and stream their own assistant/tool-use events. PreToolUse / PostToolUse hooks gate destructive operations through Nightly's approval surface. The Skill file at .claude/skills/nightly/SKILL.md ships pre-populated with the priority cascade prompt, the loop instructions, and pointers to the on-disk plan.
  • Codex CLI — installed as a Codex skill + slash command. dispatchSubAgent prefers MCP for in-session dispatch; falls back to spawning a separate codex exec --json when the sub-agent needs a fresh sandbox. Codex's native Seatbelt (macOS) / Landlock (Linux) sandboxing applies automatically — Nightly does not need to wrap it.
  • opencode — registered as a sub-agent under .opencode/agents/nightly/. The SSE server architecture is a gift: the loop just subscribes to /global/event for its event stream. POST /session/:id/fork is how specialist sub-agents are spawned with copied context.
  • Cursor — slash command + a Background Agent template. dispatchSubAgent can either invoke a Background Agent (remote VM, slow but isolated) or run inline when latency matters. Cursor's existing branch-and-PR deliverable model is the closest to Nightly's by default.
  • Antigravity — registered as a Managed Agent in the Agent Manager. Disk contract is the richest of the five: Nightly mirrors its own .nightly/runs/<ts>/tasks/<n>/ layout into Antigravity's ~/.gemini/antigravity/brain/<GUID>/ so the Agent Manager UI shows familiar walkthroughs.
A real risk: host context-window exhaustion

Because Nightly runs inside the host's session, multi-hour runs put pressure on the host's context window. Mitigation is structural: all durable Nightly state lives on disk in .nightly/, never in the host's conversation. The host can compact freely; the loop re-reads its plan, atlas, and lessons from disk on each iteration. Sub-agent dispatch via Task tool / MCP also gives each specialist its own window. The host's context is a working set, not a memory.

Headless mode — opt-in, for cron and CI

When there is no interactive session to launch from — overnight cron at 02:00, CI pipelines, scripted batches — Nightly runs in headless mode and spawns each host's CLI directly: claude -p, codex exec, opencode run. The on-disk artifact contract is identical, the loop logic is identical; only the launcher differs. Authentication still inherits from cached subscription credentials in ~/.claude/, ~/.codex/, ~/.gemini/, etc. Raw API tokens are a fallback for sandboxed CI environments without a persistent home directory — set via ANTHROPIC_API_KEY / OPENAI_API_KEY / etc. when you must, never required.

Anthropic Agent SDK metering — heads up

Starting 2026-06-15, Anthropic meters claude -p and Claude Agent SDK usage against a separate monthly credit pool tied to your subscription, distinct from interactive Claude Code usage. It's still subscription-billed (no API key required), but the credit is sub-allocated. Headless-mode Nightly users on Claude Pro/Max should account for this — either by capping headless run frequency, or by setting an opt-in ANTHROPIC_API_KEY for headless mode only. Interactive-mode Nightly (the default) is unaffected — it runs inside Claude Code's normal session and uses the normal pool.

06 · Task selection & ideation

The unsolved-on-purpose part.

Across the entire autonomous-SWE field, real autonomous selection of work — "which issue should I pick?" — is not a solved problem. The leading shipping products punt on it: humans tag (Devin / Copilot's nightly-ready label model). Nightly cannot punt that hard, because the whole product premise is unattended overnight work — but it can structure the problem so that the riskiest behavior is always the last resort.

Issue ranking — lightweight, transparent, overridable

A simple multiplicative score, recomputed each iteration. Designed to be readable in session.jsonl so any pick can be explained:

score = w_label × w_age × w_blast × w_coverage × w_recent_activity

w_label          # nightly-ready=1.5 · good-first-issue=1.2 · bug=1.1 · default=1.0
w_age            # softplus(days_open / 30), capped at 1.6 — older > older
w_blast          # 1.4 if single-file · 1.0 if multi-file · 0.5 if cross-cutting
w_coverage       # 1.3 if touched paths have >80% test coverage · 0.7 otherwise
w_recent_activity# 0.4 if a human commented in last 24h (don't step on them) · else 1.0

Three hard gates apply independently of score: an issue must (a) have a clear acceptance criterion or be transparently transformable into one, (b) not be tagged do-not-automate, and (c) not require credentials Nightly doesn't have.

Ideation — proposer, not executor

When the backlog is empty, Nightly does not spontaneously write features. The Devin migration-script failure is the warning: ungrounded autonomous "improvements" become surprise PRs that get rejected and erode trust. Instead, Nightly runs a proposer suite that emits ranked draft issues for human approval, and only escalates one to a PR if it clears a high autonomy bar.

Proposer sources

  • Dependency drift — minor + patch upgrades with green test runs (Dependabot's job, but treated as a backlog signal).
  • Coverage gaps — modules in the dependency hot path with < 60% coverage.
  • Doc-vs-code drift — signatures in README / docs/ that no longer match exported APIs.
  • TODO/FIXME audit — stale comments with associated test failures or issue links.
  • Flaky test triage — tests with a recent fail-then-pass pattern in CI history.
  • Lint debt — newly-introduced rules with autofixable violations.
  • Accessibility & perf — Lighthouse / axe-core regressions on changed pages.
  • Type holes — any / unknown at module boundaries (TS) or untyped fns (Py).
The autonomy bar for proposer-to-PR A proposed issue may be auto-executed without a human approving it only if:
  1. It is single-file and reverts cleanly.
  2. Touched paths have >90% test coverage.
  3. It is in the dep-upgrade or autofixable-lint category.
  4. The change is < 80 lines.
Everything else lands as a ranked draft issue in .nightly/runs/<ts>/proposed/ and (optionally) on the remote.

07 · Branch hygiene · check gates · uncertainty

Every change earns its review.

The deliverable model is dual: PR if there's a GitHub remote, local proposal if there isn't. Both end as a branch plus a structured artifact a human can read in five minutes. The gates below apply identically to either path — the only difference is whether the artifact is rendered into a GitHub PR body or into the morning briefing's "changes needing review" section.

Branch & worktree discipline

  • Every task gets its own git worktree, isolated by directory. Nightly's main process and the harness child process never share a working tree.
  • Branches are named nightly/<slug>-<short-ts>. main is never written to by Nightly, ever. Drained WIP parks on nightly/wip-<run-id>.
  • One PR (or proposal) per task. Multi-task PRs are forbidden — small, atomic, reviewer-friendly diffs are non-negotiable (this is the WRAP guidance from Copilot and the Cognition recipe).
  • When a GitHub remote exists, Nightly opens PRs as drafts until all checks pass; the "ready for review" flip is automatic on green, not on the agent's say-so. When there's no remote, the proposal is marked status: draft in its YAML front-matter until checks pass, then flipped to status: ready.

The done definition

A task is "done" only when, on the isolated branch:

  1. All existing tests pass locally.
  2. All new tests for new code pass (and exist — uncovered new code fails its own check).
  3. Lint, format, type-check are green locally.
  4. If a remote with CI exists: remote CI also passes. If there's no remote CI, the local check suite is the gate.
  5. The required uncertainty.md exists and is non-empty.
  6. The PR body (or proposal.md) has the structured sections (see §09).

Any of those red → artifact stays draft and the task either continues or, on drain, parks as WIP.

What "proposal" looks like in practice

When there is no GitHub remote, each task writes .nightly/runs/<ts>/tasks/<n>-<slug>/proposal.md with the same shape a PR body would have: title, summary, motivation, test plan, diff (or link to diff.patch), things that could break, things I'm not sure about, approval-needed items. The morning briefing collects these alongside any real PRs into a single "changes needing review" list; the human reviewer reads the proposal, runs nightly diff <task> for a colored view, and either nightly accept (merge into base), nightly request-changes (re-queues with feedback), or leaves it as a long-lived branch.

The refusal policy

An autonomous agent's hardest design choice is what it will not do on its own. Nightly's defaults are intentionally conservative — the Devin migration incident is what bad defaults look like — and grouped into six categories. Each refused operation writes a structured record to .nightly/runs/<ts>/proposed/approvals/<id>.md with the exact command attempted, why it was refused, and how to approve it.

Refusal is a fork, not a halt

Nightly never blocks waiting for an approval. The whole point of overnight unattended runs is that a human isn't sitting at the terminal at 03:00. When the agent hits a refusal, it forks:

  • If the refused op is optional — the task can complete without it — the task proceeds, the gap is documented in uncertainty.md, and the refused op surfaces in the briefing's "Things to approve."
  • If the refused op is required — the task cannot complete without it — Nightly rolls back the worktree to the last clean commit and parks the entire task on nightly/wip-<run-id>/<slug> with plan.md marked blocked: approval. Half-finished work is never landed as a PR or proposal.

Either way the run keeps moving: Nightly picks the next item from the priority cascade and continues. Approvals are reviewed after the fact, on the human's schedule.

  1. Destructive git. Force-push, git reset --hard on shared branches, branch deletion (-D), history rewrites (rebase -i on pushed commits, filter-repo), amending pushed commits, --no-verify / --no-gpg-sign, any direct push to main / master / release/*. Nightly only ever pushes to nightly/<slug>-<ts>.
  2. Production state. Anything that mutates a live environment — kubectl apply against a prod context, terraform apply against prod state, helm upgrade --install, deploy commands, IAM / role / permission edits, schema migrations on live databases, mass data deletion, billing-API calls in live mode, secret rotation, edits to .env or any vault / secret-manager binding (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault, 1Password, GitHub repo secrets).
  3. External communication & publishing. Sending email, posting to Slack / Discord / social channels, opening issues or PR comments in other repos, publishing packages (npm publish, pypi upload, cargo publish, docker push, gem push, helm push), calling third-party APIs that have real-world effects (payments, transactional email, infrastructure provisioning, calendar invites).
  4. Network egress to unknown domains. Nightly maintains a per-repo domain allowlist seeded from declared dependencies + AGENTS.md + prior-session traffic. Outbound calls to anything else require approval. The active allowlist is logged in .nightly/runs/<ts>/allowlist.json for audit.
  5. Scope creep. Edits outside the task's declared file scope, mass renames or moves, top-level structural changes (new git submodules, dropped lockfiles, restructured src/), CI/CD modifications (.github/workflows/*, .gitlab-ci.yml, pre-commit configs), LICENSE edits or adding GPL / AGPL-incompatible deps without an explicit policy decision, en-masse .gitignore rewrites.
  6. Bypassing test or type safety. Disabling, skipping, or deleting tests; commenting out assertions; introducing # type: ignore / # noqa / // @ts-ignore in changed paths that didn't already have them; lowering coverage thresholds; weakening type signatures to Any / unknown / any at module boundaries. The agent is allowed to add new ignores only when accompanied by a one-line justification in uncertainty.md.

How approvals work

Three approval surfaces, in order of friction and blast radius. All three are retro — they record a decision about an operation that already got refused. The next time Nightly visits the parent task, it sees the approval and retries the operation. None of them interrupt or unblock a run that's already in flight.

  • Per-op approval. nightly approve <id> from CLI (or any host session) marks one refused operation as approved. If the parent task is still queued in the current run, the loop's next visit retries it; otherwise it gets retried in the next session. Lowest friction; smallest blast radius. Best for one-off cases the agent couldn't anticipate.
  • Per-task label. Tagging the corresponding GitHub issue (or local task plan) with nightly:approved:<category> grants approval to that whole task. Same retro semantics — takes effect on the next visit. The label is recorded in plan.md for audit alongside the PR/proposal.
  • Repo policy. Editing .planning/nightly.scope.md tightens or loosens the deny list across all future runs — pre-approve dep upgrades for this repo, forbid touching certain paths entirely, etc. Nightly reads this file but never writes to it.

Tasks parked as blocked: approval are picked back up as a high-priority entry in the cascade (just below resumed in-flight plans) the moment any of the three surfaces grants approval. The cascade never blocks waiting; it just routes around blocked tasks until they're unblocked.

One category is hard-coded and not policy-overridable: destructive git operations against protected branches. Even with nightly:approved:destructive-git, Nightly will not force-push to main or rewrite shared history. That floor stays.

The uncertainty requirement

Refusal is the binary half of safety; uncertainty is the gradient half. Every task generates an uncertainty.md that the PR body or proposal links to, and Nightly's check suite refuses to mark the task ready if it's missing or empty. Required sections:

  • Things I'm not sure about — places where the agent guessed.
  • Things that could break — externally-observable risks (data, perf, API contracts).
  • Things I skipped on purpose — out-of-scope items, with reasons.
  • Approval needed for — each refused operation by id, cross-linked to proposed/approvals/<id>.md.
The Devin lesson Cognition's Devin once shipped a destructive data-migration script without flagging that it was destructive. That single incident defines the failure mode every autonomous coding agent has to design against. Nightly answers it three ways: (1) outer sandbox blocks unapproved domains/paths, (2) PR/proposal body mandates "could break" enumeration, (3) any irreversible op without an approval label aborts the task.

08 · The conclude protocol

Two thresholds. Never SIGKILL.

"Stop now" is the most dangerous instruction Nightly can receive — the obvious implementations (kill the process, drop the worktree) lose hours of work and leave the repo in a half-modified state. Borrowing from LangGraph and the killswitch.md graceful-stop pattern:

  1. User sends nightly conclude. The orchestrator flips a flag visible to the next step transition in the loop. It does not interrupt the running tool call. The current task continues to completion or to its next clean checkpoint, whichever is sooner.
  2. Nudge phase. The agent is told: "Finalize current task. No new exploration. If it can land cleanly in ≤ N minutes, land it. Otherwise stash and exit." The harness's allowed-tools list is rewritten to forbid starting new tasks. Tests, lint, and PR-creation tools stay live.
  3. Hard phase. If the nudge timeout expires, the harness's write tools are revoked — the agent can read but not edit, run tests but not push. WIP commits go to nightly/wip-<run-id> with a structured WIP.md at the root explaining what was in-flight.
  4. Report phase. The agent (with read-only tools) generates the final summary.md and Nightly's renderer produces briefing.html from the normalized event log.
  5. Teardown. Worktrees are detached but not deleted (their existence is the artifact). The session folder is committed to the repo on a nightly/session-<ts> branch.
Mid-task conclude semantics

If conclude arrives mid-task, Nightly does not halt. It tells the user "concluding — wrapping current task on nightly/<slug>, ETA ~Xm", and only after that task reaches a clean stopping point does it proceed to the report phase. The orchestrator's promise is "always drain, never abandon."

09 · The morning briefing

One HTML file. Everything that happened.

The deliverable. Generated on drain, committed to the repo, opened in a browser. Designed to be reviewed in 5–10 minutes by a sleep-deprived human with a coffee. Structure inspired by Linear changelog + Stripe Press + a really good engineering postmortem.

briefing.html · top-to-bottom
  1. Hero glance — N changes ready · M proposed issues · K tasks · runtime · LOC touched · check pass rate. The hero adapts to context: "PRs" if there's a GitHub remote, "proposals" otherwise, with the count visually identical.
  2. Changes needing review — every task's deliverable in one list. Each row has title, branch, base, diff stats, mandatory "could break" excerpt, draft / ready status, replay link, and a type badge: PR if pushed to a GitHub remote, PROPOSAL if local-only. Reviewers click through to the GitHub PR or to the embedded proposal.md renderer respectively.
  3. Proposed issues — ranked, with rationale and source signal (dep drift / coverage / a11y / etc.). Same shape regardless of remote.
  4. Things I didn't do — issues considered and skipped, with why-skipped. (Often more useful than what was done.)
  5. Things to approve — operations Nightly refused to run autonomously. Each row shows the refused op, why it was refused, what happened to the parent task (continued without it / parked as WIP on nightly/wip-...), and a one-click nightly approve <id> command. Approvals are retro — they take effect on Nightly's next visit to that task, not by interrupting a live run.
  6. Atlas deltas — what changed in the repo's structural model since last session.
  7. Lessons — new entries written to .nightly/memory/lessons.md this run.
  8. Session replay — collapsed event log, expandable per-task, with diff inlines.
  9. Next session would do — the queue Nightly would pick from if started again now. Editable; humans can re-rank.

The briefing is itself a normal HTML file — no JS framework, no server, no dashboard. Embedded styles, embedded diffs, openable offline. Same design language as this document (because that's the test: if this document is good enough to read, the briefing is too).

10 · Risks & open questions

What we don't know yet.

  • Cost variance. Long-horizon agent sessions cost real money — Cognition reports $5–$30 per resolver run for OpenHands; Devin sessions can be far more. Nightly's "until you tell me to stop" loop is open-ended by default. Open: what's the right per-session budget cap, and is it a hard kill, a nudge, or a model downgrade?
  • The "stop existing on its own" problem. If Nightly's autonomy bar for ideation is too high, it does nothing useful on empty-backlog nights and the human loses confidence. If too low, it generates surprise PRs and the human loses confidence. Open: calibrate the proposer-to-PR bar against actual repos before shipping.
  • Harness fragility. All five harnesses are evolving fast; CLI flags break. Open: integration tests against each harness on a fixed model snapshot, run on every Nightly release.
  • Sandbox parity. Only Codex offers OS-level isolation. The wrap-in-container strategy works on Linux but is awkward on macOS dev machines. Open: is the macOS Seatbelt profile reliable enough to ship as Nightly's default, or do we require Docker?
  • Atlas freshness. Devin-style wikis decay. A stale atlas is worse than no atlas. Open: regenerate on every cold start, or diff-based incremental updates?
  • Multi-repo sessions. The brainstorm assumes one repo per Nightly process. Real engineering work often crosses repos (frontend + backend + infra). Open: v2 scope or v1 must-have?
  • Concurrent task parallelism. Replit Agent 4 runs 10 concurrent tasks. Nightly could too, with worktrees making it cheap. Open: default to serial (simpler), allow opt-in parallel?
  • Authentication for the long-running case. Tokens expire. Cookies expire. Open: what's the credential-refresh strategy when Nightly is unattended for 8+ hours?

11 · A phased build plan

What we can ship in pieces.

  1. Phase 0 · Repo contract + Python workspace. uv-managed monorepo with nightly-core + per-host package stubs. pyproject.toml, ruff + Pyrefly + pytest wired behind a Makefile (make install, make check, make test). .nightly/ + .planning/ folder shape + AGENTS.md / CLAUDE.md auto-detection. Manual workflow first — write the atlas and a plan by hand, validate the contract on a real repo before any orchestrator code runs.
  2. Phase 1 · Claude Code Skill — the primary interactive host. Ship .claude/skills/nightly/SKILL.md with the loop prompt and on-disk plan pointers. Implement install() and sessionId(); defer sub-agent dispatch. Single explicit human-given task, end-to-end. Goal: prove the host-integration model.
  3. Phase 2 · Sub-agent dispatch via Task tool. Implement dispatchSubAgent mapped to Claude Code's Task tool. Specialist roles: implementer · tester · reviewer · researcher. Each runs in its own sub-agent context window. Validate the worktree + drain + uncertainty + briefing flow.
  4. Phase 3 · Issue triage + priority cascade + resumption. GitHub-issue reading, the ranking score, plan/RFC/summary resumption. Nightly is now autonomous-for-a-bounded-backlog inside Claude Code.
  5. Phase 4 · Codex + opencode skills. Second and third primary hosts. Codex unlocks the OS-sandbox story (Seatbelt/Landlock); opencode unlocks the SSE event stream for free.
  6. Phase 5 · The proposer suite. Dep drift, coverage, doc-vs-code drift, lint debt, a11y, perf. Emit ranked draft issues. Wire the autonomy bar for proposer-to-PR.
  7. Phase 6 · Cursor command + Antigravity managed agent. Secondary hosts. Cursor uses Background Agents for isolated sub-agents; Antigravity mirrors the disk contract into brain/<GUID>/.
  8. Phase 7 · Headless mode for cron + CI. Subprocess launchers (claude -p, codex exec, opencode run) with subscription-credential inheritance and optional API-token fallback. Same on-disk contract, different launch path.
  9. Phase 8 · Multi-task parallelism + concurrent worktrees. Opt-in.
